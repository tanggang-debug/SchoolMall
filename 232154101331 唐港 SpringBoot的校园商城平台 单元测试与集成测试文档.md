## 第 1 章 引言

### 1.1 编写目的

本文档用于补充SpringBoot校园商城平台详细设计样例文档中未展开的测试设计内容，明确系统在编码阶段和联调阶段需要执行的单元测试、接口测试、消息测试、数据库验证、集成测试、并发测试和异常测试。文档的目标是使测试活动能够直接服务于编码实现、微服务联调、课程演示和验收评价。

### 1.2 测试依据

| 依据类型 | 依据内容 | 对测试设计的作用 |
|---------|---------|-----------------|
| 详细设计文档 | SpringBoot校园商城平台详细设计文档 V1.0 | 确定服务职责、类设计、接口、状态机、消息、缓存和异常处理测试点 |
| 需求分析文档 | SpringBoot校园商城平台需求分析报告 V1.0 | 确定用户角色、功能需求、核心业务流程和验收目标 |
| 总体设计文档 | SpringBoot校园商城平台总体设计报告 V1.0 | 确定微服务边界、服务协作、接口风格、数据库划分和安全策略 |
| 课程实训要求 | 可实现、可演示、可分工、可验收 | 控制测试范围，优先覆盖最小可运行闭环和代表性技术点 |
| 测试规范 | 单元测试、集成测试、接口测试、异常测试方法 | 规范测试用例模板、测试执行流程和测试报告结构 |

### 1.3 被测系统概述

SpringBoot校园商城平台采用微服务架构，核心闭环包括用户登录注册、商品浏览与搜索、购物车管理、订单创建、支付处理、商户发货、用户确认收货和评价反馈。系统由 API 网关、用户认证服务、商品服务、购物车服务、订单服务、支付服务、评价服务、通知服务和统计服务等部分构成。

### 1.4 测试重点

1. **订单生命周期**是否能够按照状态机规则正确流转（待支付→已支付→待发货→已发货→已完成/已取消/售后中）。
2. **商品状态机**是否能够正确处理商品审核、上架、下架等状态变化。
3. **支付回调**是否具备签名校验和幂等处理能力。
4. **购物车与订单协作**是否能够正确处理商品库存扣减和恢复。
5. **消息队列事件**是否能够正确发布、消费、重试和幂等处理。
6. **通知服务、评价服务**是否能够与主流程正确集成。

## 第 2 章 测试目标、范围与策略

### 2.1 测试目标

1. 验证各微服务内部核心类、方法和业务规则的正确性。
2. 验证不同微服务之间通过接口和消息队列协作时的数据一致性。
3. 验证用户注册登录、商品浏览、购物车、订单创建、支付、发货、确认收货、通知和评价的端到端业务闭环。
4. 验证系统在重复请求、并发操作、支付重复回调、服务异常等情况下的可靠性。
5. 形成可复用的测试用例、测试数据、缺陷记录和测试报告模板。

### 2.2 测试范围

| 测试类别 | 覆盖对象 | 说明 |
|---------|---------|------|
| 单元测试 | 领域服务、状态机、策略类、工具类、应用服务方法 | 以类和方法为粒度，重点验证业务规则和分支逻辑 |
| 接口测试 | REST API、统一响应、错误码、权限校验 | 以接口为粒度，验证请求参数、返回结构和错误处理 |
| 消息测试 | 订单事件、支付事件、通知事件 | 验证事件生产、消费、幂等和失败重试 |
| 服务集成测试 | 订单-商品-购物车、支付-订单、订单-通知 | 验证跨服务调用和数据库状态变化 |
| 端到端测试 | 用户下单到确认收货的完整业务链路 | 验证最小可运行闭环能否完整演示 |
| 非功能测试 | 并发、幂等、安全、异常、日志 | 验证关键质量属性和可追踪能力 |

### 2.3 测试策略

| 测试阶段 | 主要活动 | 进入条件 | 退出条件 |
|---------|---------|---------|---------|
| 编码阶段 | 开发者编写并运行单元测试 | 核心类和方法完成初步编码 | 单元测试通过，核心分支有断言 |
| 服务自测阶段 | 单服务接口测试、数据库验证 | 服务可本地启动，数据库脚本可执行 | 主要接口通过，错误码符合设计 |
| 联调阶段 | 跨服务接口、消息队列和缓存联调 | 依赖服务可用或可模拟 | 核心集成流程通过 |
| 系统演示阶段 | 端到端流程、并发、异常和回归测试 | 主流程服务全部可运行 | 最小闭环可稳定演示并形成测试报告 |

### 2.4 不在本次测试范围内的内容

1. 真实第三方支付平台的全量兼容性测试，本实训采用模拟支付平台或模拟回调。
2. 真实短信运营商、对象存储的生产级压力测试。
3. 大规模生产环境容量测试和长期稳定性测试。
4. 完整的安全渗透测试，可在课程扩展阶段补充。

## 第 3 章 测试环境与测试数据

### 3.1 测试环境

| 环境项 | 推荐配置 | 说明 |
|-------|---------|------|
| 开发语言 | Java 17 或 Java 21 | 与 Spring Boot 版本保持一致 |
| 后端框架 | Spring Boot、Spring Cloud Gateway | 支持微服务接口和网关联调 |
| 数据库 | MySQL 8.x | 按服务使用独立逻辑库或独立表 |
| 缓存 | Redis 6.x 或 7.x | 用于购物车、Token、热点商品数据和分布式锁 |
| 消息队列 | RabbitMQ | 承载订单事件、支付结果和通知重试 |
| 文件存储 | 阿里OSS或本地模拟服务 | 用于商品图片、用户头像、评价图片 |
| 单元测试框架 | JUnit 5、Mockito、AssertJ | 用于类级、方法级测试 |
| 集成测试工具 | Spring Boot Test、Testcontainers、Postman、Apifox | 用于接口、服务、数据库和消息测试 |
| 覆盖率工具 | JaCoCo | 统计核心业务类测试覆盖率 |
| 构建工具 | Maven | 执行 test、package 和报告生成 |

### 3.2 测试数据分类

| 数据类型 | 准备内容 | 示例 |
|---------|---------|------|
| 用户数据 | 学生、教师、商户、管理员账号 | STUDENT、TEACHER、MERCHANT、ADMIN |
| 商品数据 | 待审核、已上架、已下架、审核拒绝的商品 | PENDING(0)、ON_SALE(1)、OFF_SHELF(2)、REJECTED(3) |
| 分类数据 | 商品分类层级 | 电子产品、服装鞋帽、图书文具等 |
| 购物车数据 | 用户购物车商品项 | cart:{userId} Hash结构 |
| 订单数据 | 待支付、已支付、待发货、已发货、已完成、已取消、售后中订单 | UNPAID(0)、PAID(1)、TO_SHIP(2)、SHIPPED(3)、COMPLETED(4)、CANCELLED(5)、REFUNDING(6) |
| 支付数据 | 支付单、支付回调日志 | INIT、SUCCESS、FAILED、REFUNDING、REFUNDED |
| 评价数据 | 商品评价、商户回复 | 评分、内容、图片 |

### 3.3 测试数据管理要求

1. 测试数据应与开发数据隔离，避免多人测试互相影响。
2. 每个测试用例应明确前置数据和预期数据变化。
3. 集成测试执行前应可初始化数据库、Redis 和消息队列。
4. 涉及并发和幂等的测试数据应使用唯一业务编号，避免历史数据污染测试结果。
5. 测试完成后应清理临时订单、支付单、通知记录和评价记录。

## 第 4 章 单元测试设计

### 4.1 单元测试原则

| 原则 | 说明 |
|-----|------|
| 独立性 | 单元测试应尽量不依赖真实数据库、Redis、消息队列或外部服务。 |
| 可重复性 | 同一测试用例在任何时间执行都应得到相同结果。 |
| 小粒度 | 一个测试用例只验证一个明确的业务点或分支。 |
| 自动化 | 单元测试应能通过 Maven 或持续集成工具自动执行。 |
| 明确断言 | 每个测试用例必须包含返回值、异常、状态或调用次数断言。 |
| 边界覆盖 | 对状态、金额、时间、数量、空值和非法值进行覆盖。 |

### 4.2 单元测试对象与覆盖目标

| 服务 | 主要测试对象 | 覆盖目标 |
|-----|-------------|---------|
| 用户认证服务 | AuthService、JwtTokenProvider、UserController | 密码校验、Token 生成、账号状态判断、用户信息更新 |
| 商品服务 | ProductService、ProductStateMachine、CategoryController | 商品创建、状态合法流转、非法流转拒绝、库存管理 |
| 购物车服务 | CartService | 添加商品、修改数量、删除商品、清空购物车 |
| 订单服务 | OrderService、OrderStateMachine | 订单创建、状态合法流转、非法流转拒绝、订单事件生成 |
| 支付服务 | PaymentService | 支付创建、支付回调处理、幂等校验、支付状态变化 |
| 评价服务 | ReviewService、ReviewController | 评价创建、评价资格校验、商户回复 |
| 通知服务 | NotificationService | 通知生成、通知查询、状态更新 |
| 统计服务 | AnalyticsService | 销售统计、热门商品、用户活跃度 |

### 4.3 单元测试用例模板

| 项目 | 内容 |
|-----|------|
| 测试用例编号 | UT-模块编号-序号，例如 UT-ORDER-001 |
| 测试模块 | 被测微服务或业务模块 |
| 测试对象 | 被测类、方法或领域组件 |
| 测试目标 | 说明该用例验证的业务规则或代码分支 |
| 前置条件 | 测试前需要准备的对象状态或 Mock 行为 |
| 输入数据 | 方法参数、Command、DTO 或事件对象 |
| Mock 对象 | Repository、Client、MQ、Redis、外部依赖等 |
| 执行步骤 | 测试执行动作 |
| 预期结果 | 返回值、异常类型、对象状态或依赖调用 |
| 断言内容 | assertEquals、assertThrows、verify 等 |
| 测试结果 | 通过 / 不通过 / 阻塞 |

```

  payment-service/
    PaymentServiceTest.java

    ReviewServiceTest.java
```

## 第 6 章 集成测试设计

### 6.1 集成测试目标

集成测试用于验证多个微服务之间的协作是否符合详细设计要求。重点关注网关路由、接口契约、服务间调用、消息事件、数据库状态、Redis 缓存一致性、异常补偿和日志追踪。

### 6.2 集成测试类型

| 类型 | 测试对象 | 示例 |
|-----|---------|------|
| 接口集成测试 | 前端/API 网关/业务服务之间的 HTTP 调用 | POST /api/orders 创建订单 |

## 第 11 章 测试通过标准与小结

### 11.1 测试通过标准

| 类型 | 通过标准 |
|-----|---------|
| 单元测试 | 核心业务类测试全部通过，核心逻辑覆盖正常、异常和边界场景 |
| 接口测试 | 接口返回格式统一，业务码、错误码、data 字段符合接口设计 |
| 集成测试 | 最小可运行闭环可以完整执行 |
| 消息测试 | 消息能够正确发布、消费、幂等处理和失败重试 |
| 数据测试 | 数据库状态、事件记录、缓存状态保持一致 |
| 并发测试 | 商品库存并发扣减等场景不产生重复业务结果 |
| 安全测试 | 未登录、越权访问等请求被正确拒绝 |
| 日志测试 | 关键业务操作记录 traceId、businessId、userId、action 和 errorCode |

### 11.2 建议优先测试的核心场景

| 优先级 | 测试场景 | 原因 |
|-------|---------|------|
| 高 | 用户登录成功与失败 | 所有业务入口依赖认证 |
| 高 | 用户创建订单 | 系统主流程起点 |
| 高 | 订单状态机合法与非法流转 | 防止业务状态混乱 |
| 高 | 支付成功后订单状态更新 | 验证核心微服务协作 |
| 高 | 商品库存扣减与恢复 | 防止库存数据错误 |
| 高 | 商户发货与用户确认收货 | 验证完整订单流程 |
| 中 | 购物车添加与修改 | 验证Redis缓存一致性 |
| 中 | 评价发布与商户回复 | 验证评价服务完整性 |
| 中 | 通知生成与查询 | 验证通知服务 |
| 低 | 统计报表、复杂售后处理 | 可作为扩展测试内容 |

### 11.3 小结

本测试文档从单元测试和集成测试两个层次对SpringBoot校园商城平台进行了完整测试设计。单元测试重点验证订单状态机、商品状态机、支付幂等、购物车操作、库存管理等服务内部核心逻辑；集成测试重点验证用户登录、商品浏览、购物车管理、订单创建、支付处理、商户发货、用户确认收货和评价发布等跨服务业务流程。通过上述测试设计，可以为后续编码阶段、接口联调阶段和课程验收阶段提供统一依据。
